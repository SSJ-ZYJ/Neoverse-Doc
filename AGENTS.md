# Agents 规则

## 代码增改准则

### 增改前

1. 所有新增内容，请在相应文件中新增注释，注释内容为新增内容的功能描述。
2. 所有增改内容，**严令禁止** 使用 **硬编码** ，请做好 i18n 本地化处理。请使用项目配置的 i18n 库进行处理，并将对应的中文文本提取。
3. 请使用项目中 package.json 定义的技术栈版本，并遵循 ESLint/Biome 的标准代码规范。如果发现未知的外部依赖，请在修改代码前并在注释中向用户确认依赖的名称及版本。
4. 请对增改的内容先去 fumadocs 官方文档中进行搜索，不要重复造轮子，避免实现重复的功能。
5. 若对 content/ 目录下的文档有增改，请务必同步更新 content/docs/zh/ 和 content/docs/en/ 目录下的文档内容。

### 增改后

1. 请整理代码，规范代码格式，确保代码质量。
2. 请检查是否存在硬编码，若存在，请及时替换为 i18n 本地化处理。
3. 在输出代码后，请提示用户运行 `bun format` 与 `bun check` 进行格式化和检查。
4. 请进行自我调试与可行性验证。
5. 如有对功能进行修改，确保修改后不会影响到其他功能的正常运行。
6. 如有对功能进行修改，需要更新相应文档、README.md、CONTRIBUTING.md 和 与之相关的文件内容，
7. 编写 Git Commit Message。
8. 请勿进行一切提交操作，包括但不限于 直接提交到远程仓库、提交到主分支等。

## 文档编写准则

### 文档命名规范

1. 文档命名请使用英文命名，且需与文档内容相关。
2. 文档命名请使用下划线分隔单词，例如：`README_en.md` 等。
3. 文档命名大小写敏感。

### 文档语言策略

1. 首期所有文档仅提供 **简体中文** 版本。
2. 后续补充英文版本时，统一使用 `_en.md` 后缀（例如 `README_en.md`、`docs/ARCH_en.md`），与中文版本同步更新。
3. **代码注释** 保留中英双语习惯，英文在上、中文在下。

### 文档内容规范

1. 命令、环境变量、文件路径等必须与代码现状一致，禁止描述未实现或已删除的能力。
2. 涉及外部依赖时，请明确版本下限。

### 文档格式规范

1. 文档格式请使用 Markdown 格式编写。
2. 所有流程图应使用 Mermaid 语法编写，并确保图表清晰、易读。
3. 文档中所有代码块请使用适当的语言标识（例如 `javascript`、`bash` 等），以便于语法高亮显示。
4. 文档中中英文之间请使用半角空格分隔，以提高可读性。
5. 文章中的英文关键字、命令、文件名等请使用反引号（`）包裹，以突出显示。
